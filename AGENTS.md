# Repository Guidelines

## プロジェクト構成とモジュール整理
現時点ではリポジトリ直下にソースコードやテスト用ディレクトリが存在しません。追加する場合は、`src/`（実装）、`tests/`（テスト）、`scripts/`（開発補助）、`assets/`（静的素材）のように役割ごとに分離してください。新規モジュールは機能単位でサブディレクトリを切り、依存関係が分かる命名にします（例: `src/telemetry/`, `src/control/`）。

## ビルド・テスト・開発コマンド
現在、ビルドやテストに関する設定ファイル（例: `package.json`, `Makefile`）は見当たりません。導入後は以下のようなコマンドを明記してください。
- `make build`: 本番用のビルドを生成
- `make test`: すべてのテストを実行
- `make lint`: コーディング規約の検査

## コーディングスタイルと命名規則
言語未確定のため共通方針のみ定義します。インデントはスペース2または4で統一し、タブは使用しません。ファイル名は小文字スネークケース（例: `control_loop.py`）か小文字ケバブケース（例: `control-loop.ts`）で統一してください。フォーマッタやリンタ（例: `black`, `eslint`, `clang-format`）を導入した場合は、その設定ファイルと実行コマンドを必ず記載します。

## テスト指針
テストフレームワークは未導入です。導入後は以下を必須とします。
- テスト配置: `tests/` 配下
- 命名規則: `test_*.py` / `*.spec.ts` などフレームワーク標準に従う
- 実行方法: 1コマンドで全テストが走ること（例: `make test`）

## コミットとプルリクエスト
このリポジトリはGitHub（`main`）で管理します。以下を基本ルールとします。
- コミットメッセージ: `type: summary` 形式（例: `feat: add calendar view`）
- 作業ブランチ: 機能ごとに `feat/...`、修正は `fix/...`（例: `feat/load-heatmap`）
- PR（任意だが推奨）: 変更概要、動作確認手順、関連Issue/メモ（あれば）

## 追加の注意点（エージェント向け）
- 自動生成ファイルや外部依存を追加する際は、再現手順とバージョンを明記してください。
- 重要な設定値は環境変数化し、サンプルは `config.example` に切り出します。
- 要件定義は「システム要件」と「ソフトウェア要件」を分けて扱います。
  - システム要件: `SYSTEM_REQUIREMENTS.md`
  - ソフトウェア要件: `SOFTWARE_REQUIREMENTS.md`
  - 毎回、どちらの要件を詰めている作業かを明確にして進めます。
- 開発プロセスはウォーターフォールで進めます。各ターンの冒頭で「現在の工程」を明示し、成果物をリポジトリに残します。
  - 外部工程: システム要件定義 → システム設計 → プログラミング → テスト → ソフトウェア受け入れ
  - システム設計（内部工程）: システム方式設計 → ソフトウェア要件定義 → ソフトウェア方式設計 → ソフトウェア詳細設計
  - 目安の成果物:
    - システム要件定義: `SYSTEM_REQUIREMENTS.md`
    - システム方式設計: `SYSTEM_DESIGN.md`
    - ソフトウェア要件定義: `SOFTWARE_REQUIREMENTS.md`
    - ソフトウェア方式設計: `SOFTWARE_ARCHITECTURE.md`
    - ソフトウェア詳細設計: `SOFTWARE_DETAILED_DESIGN.md`
    - プログラミング: 実装コード、最小限の単体テスト
    - テスト: テスト仕様/結果（結合・システム）
    - 受け入れ: 受け入れ条件の確認結果、リリース手順/成果物
- 要件（MVP）と発展案を混ぜないでください。別アイデア/将来案は `FUTURE_IDEAS.md` に退避し、要件ドキュメントは「今作るもの」だけを維持します。
- 「提案事項の確認」を求められた場合は、選択肢を列挙するだけで終えず、最善と思う推奨案（1案）を必ず明記します。
